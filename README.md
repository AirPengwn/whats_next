# What's Next for Erin

A personal, static website that helps Erin Lamberton (BS Environmental Science,
WCU '25) manage her career & graduate-school search — opportunities, trackers,
deadlines, application packets. Plain HTML + vanilla JS, hosted on GitHub Pages.

---

## ⭐ The one rule: never hand-edit `opportunities.js`

Opportunity data has a **single source of truth**:

```
opportunities.data.json   ← edit THIS (pretty JSON, full text)
        │
        │   node build.js
        ▼
opportunities.js          ← AUTO-GENERATED. Do not touch.
```

`opportunities.js` is regenerated by `build.js`. Editing it by hand will be
overwritten on the next build and reintroduces the truncation/drift bugs the
pipeline exists to prevent.

### The record schema (now holds ALL detail)

Every opportunity is a complete structured record. Existing keys are
unchanged; these were added by a one-time lossless extraction from the
(frozen, verbatim) Browse cards:

| Field | Meaning |
|---|---|
| `body` | **Short teaser** — the collapsed/summary view |
| `desc` | **Full description** — the expanded ("Details") view |
| `links` | `[{href,label}]` — every curated/labelled link on the card |
| `detail` | Ordered more-panel blocks: `{t:'h'/'p'/'row'/'contact', …}` |
| `status` | `{kind,label}` from the card's status badge (if any) |
| `metaLine` | Inline deadline/start/funding line (featured cards) |
| `cycle` | `current` or `next` (derived from the source page) |
| `deadline_date` | ISO date — the next actionable date, when parseable |
| `deadline_kind` | `fixed` · `rolling` · `contact-first` · `passed` · `unknown` — drives urgency sort/filter |
| `region` | US region bucket (8) — drives the Region filter & home quick-links |
| `expired` | `true` only for one-time/discontinued opps (no future cycle); hidden by default, never deleted |
| `badge` | Fit tier wording: Exceptional/Strong/Good/Solid, or "Reach"/"Weak fit — review" — drives the Fit filter |

The 8 Browse-page HTML files remain the **verbatim source of truth** for
existing cards (golden-gated, byte-identical). The cabinet now carries the
full structured data so dynamic views / new sections can key off it
(show/hide, sort, filter, group) without ever touching page layout.

### Update workflow

1. Edit **`opportunities.data.json`** — add/modify opportunities (full
   descriptions, no truncation). Each entry needs at least: `id`, `title`,
   `cat` (or `type`), `src`, and a description (`body` or `extbody`).
2. Run **`node build.js`**.
   - Validates the JSON. **Fails closed**: on any error it writes nothing and
     prints what's wrong.
   - Regenerates `opportunities.js` (UTF-8, no BOM).
   - Diffs against `.opp_snapshot.json` and records new/updated ids + sweep
     date in `ALL_META`.
   - Cache-busts `opportunities.js?v=…` on every page that loads it.
3. Review the diff, commit, push to your working branch, open/merge a PR.
4. GitHub Pages redeploys. Allow ~10 min for CDN/browser cache to refresh
   (everything is cache-versioned, so a hard refresh isn't normally needed).

### How the Browse pages work now (data-driven, 4 pages)

The live Browse experience is **4 lean, data-driven pages** — `erin_programs.html`
(Funded Graduate Programs), `erin_internships.html`, `erin_jobs.html`,
`erin_service.html` — each generated by `_gen_typepages.js` from proven
parts (original nav/chrome + the canonical **primary-gated** save +
the shared render engine). Each page renders straight from
`opportunities.js` (the built cabinet): locked to its type, merging
current + next cycle, **sorted by urgency**, with multi-select filters
(Cycle / Status / Fit / New + per-type subfilters) and a prominent
"✨ NEW" indicator. The global all-types view + multi-select filters live
on the **Browse All** tab of `erin_mylist.html`.

Because pages render from the cabinet, the **weekly sweep is now just
data**: add/update entries in `opportunities.data.json` → `node build.js`
→ the pages show the changes automatically (no card injection). Saving is
gated to Erin's primary device exactly as before (GET-merge-PUT).

To regenerate the 4 pages after engine/style changes:
`node _gen_typepages.js` (inputs: `_nav_block.txt`, `_theme_block.txt`,
`_ml_block.txt`, `_engine_css.txt`, `_engine_src.txt`).

**Backups:** the original 8 hand-authored Browse pages (incl. the old
`*_next.html`) are preserved in `.overhaul_backup/` (pristine) and as
`.PRE_TYPEPAGE.html`, along with pre-change snapshots of `index.html`,
`erin_mylist.html`, and the data file. The old Preserve-add markers,
`browse.pages.json`, the build.js Browse-pass, the `*_next.html` pages,
and the dev mockups/extractors have been removed (cleanup done) — the
pipeline is now purely data-driven.

## How the site works (quick map)

- **index.html** — dashboard: priority "what to do this week" + upcoming
  deadlines (driven by `priority.js`).
- **erin_mylist.html** — saved opportunities; cloud sync; **backup/restore**;
  sets the **primary device**.
- **erin_pipeline.html** — Saved → Preparing → Submitted → Awaiting → Interview
  → Outcome board (+ archived), read-only.
- **erin_search.html** — search across every opportunity and tracker.
- **erin_packet.html** — printable one-page application packet per opportunity.
- Trackers: applications, professors, references, archive, timeline, compare.
- Write: materials, writing samples, outreach.
- **styles.css** — shared theme/nav (single source for site chrome).
- **site.js** — shared: fetch-sync hook, active nav, sync chip, view-only mode.
- **priority.js** — Home priority + deadline engine.

### Data-safety model (important)

- One JSONbin holds Erin's tracker data. **Only the "primary device"
  (Erin's laptop) writes**; other devices are **view-only** (a banner shows,
  edit controls are disabled — but viewing/printing/searching still works).
- Writes are primary-gated and merge (never blind-overwrite). My List has
  Export + rolling auto-snapshots (last 5) + Restore.
- The primary-device flag is mirrored and never auto-cleared.
- `build.js` and the site never read/write Erin's cloud data or the
  primary-device flag — the data pipeline only touches static site files.
